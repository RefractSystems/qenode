# RFC-0041: Safe QOM Framework Boundaries via Type-State

**Status:** Draft
**Created:** 2026-05-17

## Summary

This RFC proposes a fundamental architectural shift in the `virtmcu-qom` framework to achieve true "Zero Unsafe" peripheral development. Instead of hiding `unsafe` blocks behind macros, we leverage Rust's type system to make invariant violations — such as accessing QEMU state outside the Big QEMU Lock or dereferencing mismatched QOM pointers — **unrepresentable at compile time**.

The architecture introduces a three-tier plan:

1. A zero-sized, `!Send` `BqlContext` proof token that replaces `DrainToken` and gates all BQL-requiring APIs.
2. A safe `ClosureTimer` abstraction that eliminates `extern "C"` callbacks from peripheral code and enforces FFI panic safety.
3. Safe, dynamically-validated QOM downcasting returning `NonNull<T>` backed by QEMU's own type system.

## Motivation

The current "Zero Unsafe" implementation relies on macros (`ffi_call!`, `new_safe`) to hide `unsafe` lexically. This removes the keyword but not the semantic hazard. Two categories of bugs remain possible:

**Category 1 — BQL threading violations.** Calling `qemu_clock_get_ns()` or `qemu_set_irq()` from a background thread causes a silent data race. The only defense today is a `debug_assert!(Bql::is_held())` inside `BqlGuarded` — absent in release builds and absent entirely from the clock/IRQ wrappers.

**Category 2 — QOM type confusion.** `deref_qom_ptr::<T>(ptr)` blindly casts any `*mut c_void` to `T`. If QEMU passes the wrong object (e.g., due to a misconfigured topology YAML), the result is silent heap corruption rather than a deterministic panic.

Enterprise SOTA engineering means **making invalid states unrepresentable**, not catching them at runtime. Rust's type system — specifically a `!Send` context token — can prove BQL ownership at compile time, converting an entire category of stochastic CI failures into immediate compiler errors.

## Detailed Design

### Tier 1: The `BqlContext` Token

`BqlContext` replaces `DrainToken` as the authoritative, zero-cost, compile-time proof that the current execution context holds the BQL. It is a zero-sized type that is non-constructible outside the framework and `!Send`, making it physically impossible to capture in a `tokio::spawn` or `crossbeam_channel` receiver closure.

```rust
use core::marker::PhantomData;

/// Zero-sized proof token: the current execution context holds the BQL.
/// Non-constructible outside the framework. `!Send` by design.
pub struct BqlContext {
    _private: (),
    _not_send: PhantomData<*mut ()>,
}

impl BqlContext {
    /// Internal: only created by the framework at the two BQL entry points:
    /// (1) the `#[qom_device]`-generated MMIO dispatch, and (2) `ClosureTimer` trampoline.
    #[doc(hidden)]
    pub(crate) unsafe fn new_unchecked() -> Self {
        Self { _private: (), _not_send: PhantomData }
    }
}
```

#### Migration from `DrainToken`

`DrainToken` (in `device.rs`) already documents that it proves "the BQL and VcpuDrain are correctly held." `BqlContext` is a direct rename with the same semantics. The two framework entry points that create it are:

1. **MMIO dispatch** (generated by `#[qom_device]`): acquires `state.drain.acquire()` → `_guard`, then creates `BqlContext::new_unchecked()` and passes it to `read`/`write`. `VcpuDrain` remains in peripheral state unchanged — it is still the teardown mechanism. `BqlContext` is the proof token.
2. **`ClosureTimer` trampoline** (see Tier 2): creates `BqlContext::new_unchecked()` after `catch_unwind` confirms we are inside the QEMU main-loop timer callback (BQL held by QEMU's invariant).

#### Updated Peripheral Trait

Only the `Peripheral` trait is updated. `MmioDevice` is already deprecated and is not touched.

```rust
pub trait Peripheral {
    /// Called during QEMU's realize phase (BQL held).
    fn realize(&mut self, ctx: &BqlContext) -> Result<(), String> { Ok(()) }

    /// MMIO read. `ctx` proves BQL held + VcpuDrain acquired.
    fn read(&self, offset: u64, size: u32, ctx: &BqlContext) -> MmioResult<'_>;

    /// MMIO write. `ctx` proves BQL held + VcpuDrain acquired.
    fn write(&self, offset: u64, value: u64, size: u32, ctx: &BqlContext);

    fn reset(&mut self) {}
    fn condvar(&self) -> &Condvar;
    fn wait_mutex(&self) -> &Mutex<()>;
}
```

#### QEMU Global Wrappers

```rust
/// BQL required: passing `&BqlContext` from a background thread is a compile error.
pub fn qemu_clock_get_ns_safe(clock_type: i32, _ctx: &BqlContext) -> i64 {
    unsafe { qemu_clock_get_ns(clock_type) }
}

pub fn qemu_set_irq_safe(irq: QemuIrq, level: i32, _ctx: &BqlContext) {
    unsafe { qemu_set_irq(irq, level) }
}
```

The `_ctx` parameter is consumed by the type checker and erased entirely in the compiled output. Zero runtime cost.

#### Upgrade of `BqlGuarded<T>` (not deprecation)

The RFC does **not** deprecate `BqlGuarded`. Replacing it with `Cell<T>` or `RefCell<T>` would be incorrect: both are `!Sync`, making any peripheral state struct containing them `!Sync`. Peripheral state is shared across multiple vCPU threads (all serialized by BQL) and must be `Sync`.

Instead, `BqlGuarded::get` and `BqlGuarded::get_mut` are upgraded to require `&BqlContext`:

```rust
impl<T> BqlGuarded<T> {
    /// Requires compile-time proof that BQL is held.
    /// Retains the runtime re-entrancy check (borrow_count) which protects
    /// against aliased borrows when BQL is temporarily yielded mid-callback.
    pub fn get<'a>(&'a self, _ctx: &'a BqlContext) -> BqlReadGuard<'a, T> { ... }

    pub fn get_mut<'a>(&'a self, _ctx: &'a BqlContext) -> BqlWriteGuard<'a, T> { ... }
}
```

The existing `debug_assert!(Bql::is_held())` inside those methods becomes a belt-and-suspenders check. The `borrow_count` re-entrancy detection (which catches aliased mutable borrows during `wait_yielding_bql`) is preserved exactly as-is.

#### Thread Safety of `BqlContext`

Because `BqlContext` is `!Send`, the following patterns become hard compiler errors:

```rust
// ERROR: BqlContext is !Send
tokio::spawn(async move { qemu_clock_get_ns_safe(QEMU_CLOCK_VIRTUAL, ctx) });

// ERROR: BqlContext is !Send — cannot appear in a crossbeam Send closure
let _ = tx.send(move || state.field.get(ctx));
```

The `VtimeIngress` Zenoh subscriber callback, which runs on a background thread, never receives a `BqlContext`. QEMU state access there is a compile error by construction.

**Correction on `VtimeIngress` execution contexts:** Both the `decode_cb` and `deliver_cb` closures passed to `VtimeIngress::new_safe` execute on the QEMU main thread under BQL — they are invoked from `deterministic_receiver_timer_cb`, an `extern "C"` timer callback. Only the innermost Zenoh subscriber (inside `SafeSubscription`) runs on a background thread; it never calls user code, only sends to a crossbeam channel. A future follow-up RFC should update `VtimeIngress::new_safe` to pass `&BqlContext` to `decode_cb` and `deliver_cb`, completing the token coverage.

### Tier 2: The `ClosureTimer`

`ClosureTimer` eliminates the need for peripheral authors to write `extern "C"` callbacks or manage raw `*mut c_void` opaque pointers for custom timers. The framework generates the trampoline internally and passes a `&BqlContext` to the closure, connecting timers into the type-safe BQL proof chain.

```rust
use core::mem::ManuallyDrop;

pub struct ClosureTimer {
    // ManuallyDrop so we control destruction order in Drop.
    inner: ManuallyDrop<QomTimer>,
    // Double-boxed: outer Box gives a stable fat-pointer address for QEMU.
    // Inner Box<dyn ...> is the type-erased closure.
    closure_ptr: *mut Box<dyn FnMut(&BqlContext) + Send>,
}

// SAFETY: the closure is Send; ClosureTimer itself is accessed only under BQL.
unsafe impl Send for ClosureTimer {}
unsafe impl Sync for ClosureTimer {}

extern "C" fn closure_trampoline(opaque: *mut core::ffi::c_void) {
    // Panicking across an extern "C" boundary is Undefined Behavior.
    // catch_unwind here, abort on panic — Fail Loudly mandate.
    let result = core::panic::catch_unwind(core::panic::AssertUnwindSafe(|| {
        // SAFETY: opaque is the stable address of the outer Box, valid until Drop.
        // QEMU guarantees the BQL is held during all timer callbacks.
        let cb = unsafe { &mut **(opaque as *mut Box<dyn FnMut(&BqlContext) + Send>) };
        let ctx = unsafe { BqlContext::new_unchecked() };
        cb(&ctx);
    }));
    if result.is_err() {
        std::process::abort();
    }
}

impl ClosureTimer {
    /// Creates a new timer that fires `f` with a `&BqlContext` at the given virtual time.
    pub fn new<F>(clock_type: i32, f: F) -> Self
    where
        F: FnMut(&BqlContext) + Send + 'static,
    {
        // Double-box: Box<F> → type erase → Box<dyn FnMut(...)>
        // The outer Box::into_raw gives a stable address for the QEMU opaque pointer.
        let closure: Box<Box<dyn FnMut(&BqlContext) + Send>> =
            Box::new(Box::new(f));
        let closure_ptr = Box::into_raw(closure);

        // SAFETY: closure_ptr is valid until Drop calls Box::from_raw.
        let inner = QomTimer::new_safe(
            clock_type,
            closure_trampoline,
            closure_ptr as *mut core::ffi::c_void,
        );

        Self { inner: ManuallyDrop::new(inner), closure_ptr }
    }

    /// Arms the timer to fire at `expire_ns` (virtual nanoseconds).
    pub fn arm(&self, expire_ns: i64) {
        self.inner.mod_ns(expire_ns);
    }

    /// Disarms the timer without destroying it.
    pub fn disarm(&self) {
        self.inner.del();
    }
}

impl Drop for ClosureTimer {
    fn drop(&mut self) {
        // Order is load-bearing:
        // 1. Delete + free the QEMU timer (no more callbacks will fire after this).
        // 2. Only then reconstruct and drop the closure Box.
        // Reversing this order would create a dangling opaque pointer.
        unsafe { ManuallyDrop::drop(&mut self.inner) };
        unsafe { drop(Box::from_raw(self.closure_ptr)) };
    }
}
```

Peripheral usage:

```rust
// realize():
let has_data = Arc::clone(&self.has_data);
let cond = Arc::clone(&self.cond);
self.poll_timer = Some(ClosureTimer::new(QEMU_CLOCK_VIRTUAL, move |ctx| {
    let now = qemu_clock_get_ns_safe(QEMU_CLOCK_VIRTUAL, ctx); // compile-time BQL proof
    has_data.store(true, Ordering::Release);
    cond.notify_all();
    drop(now);
}));
self.poll_timer.as_ref().expect("just set").arm(next_deadline_ns);
```

No `extern "C"`, no raw pointer casting, no manual opaque recovery in peripheral code.

### Tier 3: Type-Validated QOM Downcasting

`deref_qom_ptr::<T>(ptr)` and `opaque_to_state::<T>(ptr)` in `timer.rs` perform blind casts. A misconfigured topology YAML pointing the wrong QOM link at a peripheral causes immediate heap corruption with no diagnostic.

We replace these with a validated downcast backed by QEMU's own `object_dynamic_cast` (already declared in `qom.rs:139`):

```rust
/// Downcasts a raw QOM pointer using QEMU's dynamic type system.
/// Returns `None` on null or type mismatch; never blindly reinterprets memory.
///
/// The caller must still write `unsafe { ptr.as_mut() }` to dereference —
/// lifetime binding is the caller's responsibility. This is intentional:
/// the framework validates the type; the caller reasons about the lifetime.
pub fn dynamic_cast_qom<T: QomDevice>(ptr: *mut core::ffi::c_void) -> Option<NonNull<T>> {
    if ptr.is_null() {
        return None;
    }
    // SAFETY: ptr is non-null; object_dynamic_cast checks QEMU's type chain.
    let result = unsafe {
        crate::qom::object_dynamic_cast(ptr as *mut crate::qom::Object, T::QOM_TYPE_NAME.as_ptr())
    };
    NonNull::new(result as *mut T)
}
```

**`QOM_TYPE_NAME` generation:** The `#[qom_device]` macro (RFC-0023) is extended to emit:

```rust
impl QomDevice for MyPeripheralQEMU {
    const QOM_TYPE_NAME: &'static core::ffi::CStr = c"my-peripheral";
}
```

This value is the exact string registered with `type_register_static` — the macro already has it, so no new information is required from the peripheral author.

**Why `NonNull<T>` not `&'a mut T`:** An unconstrained lifetime `'a` in the return type is a soundness hole — the caller can infer `'static`, giving a reference that outlives the QOM object. `NonNull<T>` is honest: the pointer is typed and non-null, but the caller must supply the correct lifetime at the dereference site where they have the contextual information to do so correctly.

**Usage with Fail Loudly:**

```rust
let dev = dynamic_cast_qom::<MyPeripheralQEMU>(raw_ptr)
    .expect("FATAL: QOM type mismatch — topology YAML is misconfigured");
// Only now, with a typed NonNull, dereference within the known-valid scope:
let dev_ref = unsafe { dev.as_mut() };
```

## Relationship Between `BqlContext` and `VcpuDrain`

These serve distinct, complementary roles and remain separate:

| | `VcpuDrain` | `BqlContext` |
|---|---|---|
| **What it is** | RAII teardown mechanism | Zero-sized proof token |
| **Lives in** | Peripheral state struct | Stack, not stored |
| **Proves** | Active vCPU count is tracked | BQL is held right now |
| **Created by** | Peripheral `new()` | `#[qom_device]` dispatch or `ClosureTimer` trampoline |
| **Destroyed by** | `Drop` on peripheral teardown | Falls off stack |

In the `#[qom_device]`-generated MMIO dispatch the two work together:

```rust
// Generated code (schematic):
let _drain_guard = qemu_state.drain.acquire(); // VcpuDrain: count++
let ctx = unsafe { BqlContext::new_unchecked() }; // proof token
peripheral.read(offset, size, &ctx);             // both invariants satisfied
// _drain_guard drops here: count--
```

`VcpuDrain` does not change. Its `wait_for_drain` / `Drop` behavior is identical to today.

## Migration Checklist for Peripheral Authors

For each peripheral in `hw/rust/`:

- [ ] `fn read(&self, offset, size, token: &DrainToken)` → `fn read(&self, offset, size, ctx: &BqlContext)`
- [ ] `fn write(&self, offset, value, size, token: &DrainToken)` → same rename
- [ ] `fn realize(&mut self) -> Result<...>` → `fn realize(&mut self, ctx: &BqlContext) -> Result<...>`
- [ ] All `state.field.get()` / `state.field.get_mut()` on `BqlGuarded<T>` → pass `ctx`
- [ ] `qemu_clock_get_ns_safe(clock_type)` → `qemu_clock_get_ns_safe(clock_type, ctx)`
- [ ] Any standalone `extern "C"` timer callback → replace with `ClosureTimer::new(...)`
- [ ] Remove any `#[allow(clippy::not_unsafe_ptr_arg_deref)]` on now-deleted raw-pointer timer constructors
- [ ] `deref_qom_ptr::<T>(ptr)` → `dynamic_cast_qom::<T>(ptr).expect("...")` + `unsafe { ptr.as_mut() }`

The `#[qom_device]` macro generates the MMIO dispatch, so the macro is updated once; all peripherals inherit the `ctx` thread-through at their call sites only.

## Drawbacks

- **Refactoring scope:** Every peripheral must thread `&BqlContext` through its MMIO, `realize`, and timer callbacks. This is mechanical but broad.
- **Hard errors are intentional:** Peripheral code that accessed QEMU globals from background threads will fail to compile. This is the goal, not a regression.

## Alternatives

- **Keep `ffi_call!` macros:** Hides `unsafe` lexically but leaves all semantic hazards intact. Rejected.
- **Runtime assertions only (`assert!(Bql::is_held())`):** Zero-cost approach is possible but wrong here — `assert!` through an `extern "C"` frame is UB (unwind across FFI); `process::abort()` is safe but still runtime-only. The compile-time approach is strictly better.
- **Replace `BqlGuarded` with `Cell`/`RefCell`:** `Cell<T>` and `RefCell<T>` are `!Sync`. Peripheral state structs must be `Sync` because the BQL serializes access from multiple vCPU threads simultaneously. This replacement would require `unsafe impl Sync` on every peripheral, which is strictly worse than the current design.
- **Global Rust `Mutex`:** Deadlocks with QEMU's BQL. Rejected.

## Unresolved Questions

- **`VtimeIngress::new_safe` callback signatures:** Both `decode_cb` and `deliver_cb` execute under BQL (inside `deterministic_receiver_timer_cb`). A follow-up RFC should update their signatures to `FDecode: Fn(&BqlContext, &str, &[u8]) -> Option<T>` and `FDeliver: FnMut(&BqlContext, T)`, completing the compile-time BQL proof across all framework callback types.
- **Legacy `VtimeIngress::new` (raw function pointer API):** Can be removed once all peripherals migrate to `new_safe`. Should be tracked as a deprecation in the same follow-up.
