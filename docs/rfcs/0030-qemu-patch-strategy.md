# RFC-0030: QEMU Patch Strategy and "No Fork" Policy

## Summary
This RFC formalizes the policy of maintaining VirtMCU core enhancements and fixes as a set of managed patches against upstream QEMU, rather than maintaining a hard-forked Git repository.

## Motivation
VirtMCU relies on a specific version of QEMU (currently 9.x) with targeted modifications to the TCG loop, clock synchronization, and the `arm-generic-fdt` machine. Maintaining a hard-fork of QEMU is a massive maintenance burden:
- Upstreaming fixes from a fork back to QEMU is difficult.
- Re-basing a fork onto new QEMU releases (e.g., moving from 9.1 to 9.2) often results in complex merge conflicts.
- It obscures exactly which changes VirtMCU has made to the core emulator.

By using a "No Fork" patch-based strategy, we ensure that our changes are surgical, documented, and easily portable across QEMU versions.

## Guide-level explanation
As a developer, you should treat the `third_party/qemu` directory as a read-only, ephemeral artifact generated by the build system.

1. **Do not commit changes to `third_party/qemu`**: Any manual changes will be wiped during the next `make bootstrap` or `cargo run -p virtmcu-cli -- setup patch-qemu`.
2. **Adding a core fix**: If you need to modify QEMU's C source code:
   - Make your changes in `third_party/qemu` to test them.
   - Use `git diff` inside that directory to generate a `.patch` file.
   - Save the patch in the `patches/` directory.
   - Register the patch in `tools/virtmcu-cli/src/setup/patches.rs` (or the relevant metadata file).
3. **Automated Application**: The build system will automatically apply these patches in order whenever the environment is initialized.

## Reference-level explanation
- **Storage**: All patches are stored in the root `patches/` directory.
- **Enforcement**: A CI lint (`check-qemu-dirty`) compares the state of `third_party/qemu` against the base version plus the `patches/` directory. If they differ and no new patch was added to the repo, the build fails.
- **Patch Format**: Standard unified diff format (`git format-patch` or `diff -u`).
- **Atomic Setup**: The command `virtmcu-cli setup patch-qemu` is responsible for:
  1. Resetting the submodule to the pinned `QEMU_VERSION` in `BUILD_DEPS`.
  2. Iterating through the `patches/` directory in alphanumeric order.
  3. Applying each patch via `patch -p1`.
  4. Running the Zenoh-hook injection script (`apply_zenoh_hook.py`).

## Drawbacks
- **Compile Time**: Applying patches and re-configuring QEMU can be slower than pulling a pre-built forked container image. (Mitigated by the `ci-build-third-party` Docker caching strategy).
- **Conflict Management**: If two developers modify the same line in QEMU, the patch application will fail. This forces early resolution of architectural conflicts.

## Rationale and alternatives
- **Alternative: Hard Fork**: We could maintain `google/qemu-virtmcu`. This would allow standard Git workflows but would make "upstreaming" our changes to the official QEMU project significantly harder and would hide our changes in a sea of 1.5 million lines of QEMU code.
- **Alternative: Binary-only distribution**: We could distribute pre-compiled QEMU binaries. This violates the project's goal of being a "Transparent White-Box Simulator" and would make debugging core TCG issues impossible for contributors.

## Unresolved questions
- How do we handle patches that are specific to a certain host architecture (e.g., fixes only required for macOS/Apple Silicon)?
- At what point does a set of patches become "too large," necessitating a move back to a fork? (Target limit: < 50 discrete patches).
